端口的粒度与命名：一段有目的的对话

第 1 课你分清了左右。这一课回答评审时最常被追问的两句： 这个接口算不算一个端口？以及它该叫什么？

你在地图上的位置：本课钉死速查地图 · 核心定义里 「端口 = 有目的的对话」与「典型 2–4 个」两格，并补上端口对话速查。 上一课唯一软点——「通知型」被驱动端口——课首复验。

回顾检查

两题：一道钉「仅仅通知」；一道钉「谁先开口」。选项字数对齐，别靠长短猜。

端口是什么：一段有目的的对话

A port identifies a purposeful conversation. … The protocol for a port is given by the purpose of the conversation between the two devices. 端口标识一段有目的的对话。……端口的协议，由双方对话的目的给出。 —— Alistair Cockburn, "Hexagonal Architecture" (2005)

操作系统端口、电子设备插口——凡遵守协议者皆可插入。协议谈的是目的（存订单、算税、通知订阅者）， 不是电缆材料（MySQL、REST、SMTP）。技术只出现在适配器里。

Cockburn 近年演讲里的命名句式：For_doing_something （见 Budapest 2023 幻灯）。 DDD 课你用的 OrderRepository 同样合格——名词角色名，意图清晰。 两种风格都行；按技术命名一律不合格。

粒度：两极端都错，中间那一小撮才对

What exactly a port is and isn't is largely a matter of taste. At the one extreme, every use case could be given its own port, producing hundreds of ports… Alternatively, one could imagine merging all primary ports and all secondary ports so there are only two ports… My selection tends to favor a small number, two, three or four ports… 端口划到哪是品味问题。一极端：每个用例一个端口，上百个；另一极端：左合并成一个、右合并成一个，只剩两个。 我倾向少量——两、三、四个。 —— 同上，"How Many Ports?"

过细 · 按用例爆炸

  • PlaceOrderPort
  • CancelOrderPort
  • AmendOrderPort
  • …上百个「端口」

对话目的其实都是「操作订单」——拆碎了只剩函数清单。

过粗 · 按左右两桶

  • LeftSideApi（所有驱动）
  • RightSideApi（所有被驱动）

丢掉了「对话目的」——评审时说不清系统在跟外界谈什么。

合适 · 按自然对话（Cockburn 天气系统例：4 个）

  • 采集气象信息 · 管理 · 通知订阅者 · 读写订阅者数据
  • 咖啡机例：用户 · 配方库 · 出货装置 · 投币盒

判据：换掉一种技术，这段对话还在吗？在 → 它是端口；不在 → 那是适配器细节。

第 1 课提过「六边形 ≠ 6 个端口」——六只是画图留位。真正该数的是有目的的对话，通常 2–4 段。

一条绿色断言：一对话，两技术

code/lesson2_ports.cc（clang / -std=c++20 -Wall -Wextra 零告警）。 应用只认识 ForStoringOrders；内存适配器与「伪文件」适配器先后插入——应用源码一字未改：

struct ForStoringOrders {                 // purpose, not tech
  virtual void save(const OrderRecord&) = 0;
  virtual OrderRecord load(int id) = 0;
};

struct ForRecordingAudit {                // merely notify — no answer
  virtual void record(const std::string& event) = 0;
};

// Adapter A and Adapter B both implement ForStoringOrders.
// Had we named ports ForMysql / ForFile, the app would depend on
// two conversations for one purpose — the smell this lesson rejects.
PlaceOrderApp app_a(in_memory_store, audit);
PlaceOrderApp app_b(fake_file_store, audit);

ForRecordingAudit 是通知型被驱动端口：应用开口、外界接收、不要答案—— 与第 1 课 CustomerNotifier 同槽，只是换了领域词，避免靠记题蒙对。
