配置式依赖：适配器在外部决定

You get to replace production connections with test harnesses, and back again, without changing the source code. 你可以把生产连接换成测试线束，再换回来——不必改应用源码。 —— Alistair Cockburn, Budapest 2023 · Benefits

2005 原文把这件事挂在 Dependency Inversion / Injection 上： 高模块不依赖低模块，双方依赖抽象；具体适配器可替换。近年他给接线角色起了名字： configurator（配置器）。

同一角色在 DI 文献里叫 Composition Root（组装根）： 「应用中（最好唯一的）把模块组装到一起的位置——越靠近入口越好。」 控制台程序里，那就是 main；测试里，那就是测试 setup。两者特权相同。

组装根：唯一知道具体类型的地方

合格 · 组装根接线

auto store = std::make_unique<PostgresOrderStore>();
PlaceOrderApp app(*store);          // port ref in
int id = app.place("SKU-1");        // use at driving port

Postgres… 只出现在 main / 测试里。应用源码搜不到。

不合格 · 应用自己 new

class PlaceOrderApp {
  PostgresOrderStore store_;   // tech inside
  PlaceOrderApp() : store_() {}
};

换替身必须改应用源码 → 配置式依赖宣告死亡。

Cockburn 展示了两种接线手法：构造函数/setter 注入，以及 broker 查找。本课与 mission 一致：先手工构造函数注入，不上 DI 容器。 容器若日后出现，也只能待在组装根——别处不许 #include 容器头。

C++：组装根拥有，应用借用

code/lesson3_composition.cc（clang / -std=c++20 -Wall -Wextra 零告警）。 同一 PlaceOrderApp 类型，被两个组装根分别插入生产形适配器与内存替身：

class PlaceOrderApp final : public ForPlacingOrders {
 public:
  explicit PlaceOrderApp(ForStoringOrders& store)  // borrow, don't own
      : store_(store), next_id_(1) {}
  // …
 private:
  ForStoringOrders& store_;
};

void compose_production() {
  auto store = std::make_unique<PostgresOrderStore>();  // root OWNS
  PlaceOrderApp app(*store);
  assert(app.place("SKU-PROD") == 1);
}

void compose_test() {
  InMemoryOrderStore store;   // test entry = also a composition root
  PlaceOrderApp app(store);
  assert(app.place("SKU-TEST") == 1);
}

口诀：组装根拥有，应用借用。应用构造函数收端口引用（或非拥有指针）， 绝不 new 适配器。适配器寿命由组装根保证 ≥ 应用使用期——这是 C++ 里把 「可配置依赖」钉死的方式，不是风格偏好。

顺带复验第 2 课软点：PostgresOrderStore 这个技术类名 出现在适配器 / 组装根上完全合格；若它出现在端口 typedef 或应用成员类型上，才坏。
