六边形架构 · 第 1 课

从分层到六边形：你已经会一半

DDD 课教会了你右半边（仓储端口、适配器、依赖倒置）。这一课补上分层图里根本不存在的左半边：UI、测试、定时任务——它们也是适配器。

你在地图上的位置：本课覆盖速查地图的「全景图」与「三个常见误读」。 你从 DDD 课带来的全部战术工具（实体/值对象/聚合/领域服务）原封不动地住进六边形内部。

回顾检查

先热身两题，考的是这门课要踩的地基（DDD 课第 7、8 课内容）。

发明它是为了什么

Allow an application to equally be driven by users, programs, automated test or batch scripts, and to be developed and tested in isolation from its eventual run-time devices and databases. 让应用可以被用户、程序、自动化测试或批处理脚本平等地驱动；并且脱离最终运行时的设备与数据库，独立开发、独立测试。 —— Alistair Cockburn, "Hexagonal Architecture" (2005)

注意 intent 里没有一个字提「分层」或「解耦」。它提的是平等（equally）：测试和用户是同级驱动者。这就是六边形与你熟悉的分层图的分水岭。

把 UI 从顶层拽下来

你在 DDD 课画的分层图是一维的：UI 在上，DB 在下。六边形把这张图对折——上下两端捏到一起，全都放到「外面」：

左：驱动侧（发起对话）· 右：被驱动侧（被调用）。适配器之间没有层级，上下方位无意义。

被驱动侧（右）你已滚瓜烂熟：OrderRepository 是端口，SqlOrderRepository 是适配器，依赖倒置。新增量全在左边：应用自己也经由端口被外界调用——应用层用例的对外 API 就是驱动端口，而 REST controller 只是插在这个端口上的一个适配器。换成 CLI、换成测试，端口不变。

两侧唯一的判据：谁发起对话

A primary actor is an actor that drives the application (takes it out of quiescent state to perform one of its advertised functions). A secondary actor is one that the application drives, either to get answers from or to merely notify. 主动（驱动）参与者：把应用从静止态唤醒、让它执行既定功能的一方。被动（被驱动）参与者：被应用调用的一方——要么向它要答案，要么只是通知它。 —— 同上

判断口诀：谁先开口。发起调用的在左边；被调用的在右边。数据库「重要」不代表它驱动谁——它永远在等应用开口，所以是被驱动侧。

测试是一等适配器——C++ 证据

下面是 code/lesson1_check.cc 的核心（clang / -std=c++20 -Wall -Wextra 零告警，断言全绿）。看两个不同的驱动者插进同一个驱动端口：

// Driving port: the conversation the app OFFERS. Named after intent, not tech.
struct PlaceOrder {
  virtual int execute(const PlaceOrderCommand&) = 0;
  virtual ~PlaceOrder() = default;
};

// Driver #1: an HTTP-ish controller — just a translator into the port's language.
int fake_http_controller(PlaceOrder& app, const std::string& sku, int qty) {
  return app.execute(PlaceOrderCommand{sku, qty});
}

// Driver #2: the test itself — same port, same rank, no controller needed.
const int id_from_test = app.execute(PlaceOrderCommand{"SKU-42", 3});
const int id_from_http = fake_http_controller(app, "SKU-7", 1);

右侧同理：测试用的 InMemoryOrderRepository 和生产的 SqlOrderRepository 插同一个被驱动端口——测试替身不是二等公民，它就是一个适配器。而 PlaceOrderService 全文没有出现 HTTP、SQL、SMTP、test 任何一个词。

三个常见误读（评审时的高频弹药，详见速查地图）：

  • 「六边形 = 6 个端口」？否——六只是给端口留画图位置，摆脱一维分层图；Cockburn 自述典型端口数 2–4 个。
  • 「分层架构换个画法」？否——只有里/外之分，适配器之间无层级。
  • 「UI 层在最上面」？六边形里 UI 与 DB 地位对称，都在外面。
