对比评审：别猜流派，追箭头

Hexagonal、Clean、Onion 有家族相似性，但观察轴不同。 评审时先找可执行证据，最后才给架构贴名字。

你在地图上的位置：前四课已经让你会画、会接、会测。 这一课把这些能力压成一套 架构评审速查： 能指出业务泄漏在哪里，也能说清是否值得使用这些模式。

回顾检查

两题专门回收上一课的软点：主目标是 S2；测试线束不等于生产驱动。

先下结论：三兄弟不是三选一

同一个系统可以同时被合理地称为 Hexagonal、Clean 和 Onion。 它们都把业务政策放在内侧、技术机制放在外侧，并让源码依赖朝内。

差别主要是作者想让你看见什么：Hexagonal 看跨边界的对话； Clean 看政策层级与数据过界；Onion 看领域模型居中及基础设施外置。

视图最想让你看见图上核心评审时追问
Hexagonal 应用与外界的对话；驱动/被驱动不对称 应用 + ports；外侧是可替换 adapters 谁先开口？协议属于谁？能否无 UI、无 DB 做 S2？
Clean 政策与机制的层级；数据跨圈转换 Entities → Use Cases；外圈是 Interface Adapters / Frameworks 源码依赖是否只朝内？内圈是否提到外圈名称或数据格式？
Onion Domain Model 居中；基础设施与测试在外 领域模型；仓储等接口靠近应用核心 领域是否只依赖自身？数据库实现是否留在边缘？
传统分层 按职责降低认知负担 Presentation → Domain → Data Source 底层依赖是否把领域拖向数据库？这个复杂度值得反转吗？

出处不是二手总结：Cockburn 定义 ports / adapters 与隔离测试； Robert C. Martin 的 Dependency Rule 要求源码依赖只朝内；Jeffrey Palermo 的 Onion 原文 把 Domain Model 放在中心；Fowler 的 Presentation–Domain–Data 同时说明传统分层的价值与典型依赖方向。

别把传统分层做成稻草人

传统分层不是「错误架构」。Fowler 强调它能缩小每次思考的范围，且对小型系统很有效。 真正的风险是常见的传递耦合：

Presentation -> Domain -> SqlOrderRepository -> PostgreSQL
                  ^
                  business rules cannot run without the data mechanism

一旦在 Domain 与 Data Source 之间放入由内侧拥有的抽象，并把实现翻到外侧， Fowler 也指出这通常会被称为 Hexagonal。分界不在目录名，而在源码依赖箭头。

评审算法：六步拿证据

  1. 选一个业务变化。例如「库存不足拒单规则改变」，不要从文件夹树开始。
  2. 追编译依赖。业务代码是否引用 SQL、HTTP、框架 DTO、全局容器？传递依赖也算。
  3. 圈出边界对话。谁驱动应用？应用又需要谁回答或收通知？端口名是否表达业务意图？
  4. 做替换测试。REST 换 CLI、Postgres 换内存替身，是否只改组装根与适配器？
  5. 做 S2 测试。业务规则能否无 UI、无真库，经驱动端口自动回归？
  6. 最后讨论取舍。复杂领域与长寿系统收益大；简单 CRUD 可能只需清楚分层，不必堆接口。

评审句式：不要说「这不够 Clean」。说： 「PlaceOrder 直接构造 PgConnection，导致库存规则无法在 S2 隔离回归； 建议把持久化对话抽成内侧拥有的端口，具体连接移到组装根。」
