回顾检查

两题：一道钉配置式依赖；一道钉组装根位置。选项字数对齐。

意图原文：没有 UI、没有数据库也能跑

Create your application to work without either a UI or a database so you can run automated regression-tests against the application… 让应用在既没有 UI 也没有数据库的情况下仍能工作， 以便对它跑自动化回归测试…… —— Alistair Cockburn, "Hexagonal Architecture" (2005) · Intent

这不是「多写几个 mock」的工程技巧——这是模式的定义性要求。 做不到孤立回归，就还没建成六边形。

左右不对称：自然配对

驱动侧 ← 测试线束

测试是第一个用户（Juan）。它通过驱动端口唤醒应用—— 与 REST/CLI 走同一扇门。Cockburn 原文自然例子：FIT / FitNesse。

被驱动侧 → 测试替身

答查询（stub）或记录通知（spy）。技术名可以叫 InMemoryOrderStore——那是适配器。原文自然例子：mock 数据库。

The natural test adapter to substitute for a primary actor is FIT… for a secondary actor such as a database is a mock… 替主参与者 → FIT；替次参与者（数据库）→ mock。 —— 同上，"Use of FIT and Mocks / "Left" and "Right" sides

搞反了就废：用 mock「假装」驱动侧、却让测试去摸真实 SMTP——你测的是电缆，不是对话。

开发顺序矩阵（S1–S5）

合著者 Juan M. Garrido de Paz 按 Cockburn（Agiles2020 等演讲）整理的阶段表：

阶段驱动侧六边形被驱动侧
S1测试用例硬编码—
S2 ★测试用例真实测试替身
S3真实适配器真实测试替身
S4测试用例真实真实适配器
S5真实真实真实

S2 之后，六边形（业务逻辑）就算做完了——可以在与真实设备隔离的情况下测通。 Juan 原文：This is the main goal of Hexagonal Architecture. S3 换左（上 CLI/HTTP）、S4 换右（接真库）、S5 全真——都是在 S2 立住之后加电缆。

不必纠结这条测试该叫「单元」还是「集成」。Juan 明确把测试分类术语踢出范围： 重要的是业务能否孤立被测。

一条绿色断言：S2 现场

code/lesson4_isolation.cc（clang / -std=c++20 -Wall -Wextra 零告警）。 测试组装根插入两个被驱动替身，再从驱动端口进攻：

InMemoryOrderStore store;     // driven double
SpyNotifier notifier;         // driven spy (merely-notify port)
PlaceOrderApp app(store, notifier);

const int id = app.place("SKU-42");   // drive ForPlacingOrders
assert(id == 1);
assert(store.load(1).sku == "SKU-42");
assert(notifier.placed_ == std::vector<int>{1});

没有 Postgres，没有 SMTP，没有 REST——却断言了业务规则与「通知确实发出」。 这就是矩阵里那一格高亮的 S2。
