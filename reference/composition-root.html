组装根速查

谁有权知道 PostgresOrderStore？——只有组装根。 定义对齐 Cockburn 的 configurator 与 Seemann 的 Composition Root。

一句话

组装根（Composition Root / configurator）是应用入口附近、 唯一把具体适配器插进端口的地方。六边形内部只见端口，不见技术类名。

谁在哪一侧知道什么

角色可以知道绝不可以
应用（六边形内） 端口接口（ForStoringOrders&） new PostgresOrderStore、#include <libpq>、服务定位器
适配器 自己实现的端口 + 一种技术 替应用做用例编排；持有别的适配器当「朋友」
组装根（main / 测试 setup） 全部具体适配器类型，并完成接线 业务规则（算税、校验订单……那是应用的事）

C++ 所有权口诀

组装根拥有 · 应用借用

  • unique_ptr<PostgresOrderStore> 或栈对象在 main
  • 应用构造函数收 ForStoringOrders&
  • 适配器寿命 ≥ 应用使用期

应用自己 new 适配器

  • 技术名漏进六边形内部
  • 测试无法在不改源码的情况下换替身
  • 与 Cockburn intent「equally driven / swapped」直接冲突

两种接线方式（Cockburn 2023）

  1. 构造函数 / setter 注入（DI）——本课默认；C++ 手工组装首选。
  2. Broker / 依赖查找（Lookup）——应用向 broker 要端口实现；broker 仍由组装根配置。 别把 lookup 撒进领域对象。

Mission 约束：先掌握手工组装，不上 DI 框架。框架若出现，也只能待在组装根里 （Seemann：容器只被 Composition Root 引用）。

30 秒自检

  1. 搜应用源码：有没有 Postgres / libpq / Kafka？有 → 泄漏。
  2. main（或测试 setup）之外还有没有 make_unique<某适配器>？有 → 组装根不唯一。
  3. 换测试替身要不要改应用 .cc？要 → 依赖不是配置式的。
