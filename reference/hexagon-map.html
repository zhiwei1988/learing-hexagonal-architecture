六边形架构速查地图

一页看懂端口与适配器。定义一律以 Cockburn 2005 原文为准。

全景图

驱动侧（左）· Driving

发起对话的一方 —— 把应用从静止态唤醒

  • REST/gRPC controller
  • CLI、GUI
  • 自动化测试
  • 定时任务 / 消息消费者

六边形内部 · The Application

领域 + 应用编排，技术无关

  • 驱动端口：应用提供的对话 —— 用例边界（如 PlaceOrder）
  • 领域模型：实体 / 值对象 / 聚合 / 领域服务（DDD 课全套住这里）
  • 被驱动端口：应用需要的对话（如 OrderRepository、CustomerNotifier）

被驱动侧（右）· Driven

被应用调用的一方 —— 提供答案或接收通知

  • SQL 仓储适配器
  • SMTP / 短信适配器
  • Kafka 生产者适配器
  • 内存假仓储（测试替身）

核心定义

术语定义判据 / 备注
端口 Port 一段「有目的的对话」的边界接口。名字取自操作系统端口：凡遵守协议者皆可插入。 按意图命名（OrderRepository），不按技术命名（MySqlGateway）。典型应用只有 2–4 个端口。
适配器 Adapter 在端口协议与某种具体技术之间做双向翻译的组件。 一个端口可插多个适配器。测试替身也是适配器，与生产适配器同级。
驱动侧 Driving
（primary）		 发起对话、把应用从静止态唤醒的参与者及其适配器。 判据只有一条：谁发起。UI、测试、cron 全在此侧。
被驱动侧 Driven
（secondary）		 被应用调用的参与者：要么给答案（仓储），要么收通知（邮件、事件）。 端口由应用拥有并定义，适配器实现它 —— 即 DDD 课的依赖倒置。
配置式依赖
Configurable Dependency		 用哪个适配器，在应用外部（组装根）决定并插入，内部只见端口。 第 3 课展开。C++ 里即「构造函数收端口引用/智能指针」。

与 DDD 三层的映射

DDD 课的分层六边形里的位置
领域层六边形内部（核心）
应用层（编排/用例）六边形内部；其对外 API 即驱动端口
基础设施层拆散了：全变成外侧的各个适配器，彼此无层级
表现层 / UI（分层图的顶层）降级为驱动适配器——与 DB 适配器地位对称

三个常见误读

  • 「六边形 = 6 个端口」——否。六只是画图方便：摆脱一维分层图，给端口留位置。Cockburn 自述典型端口数是 2–4 个。
  • 「就是分层架构换个画法」——否。六边形只有里/外之分，适配器之间没有层级，上/下方位没有意义。
  • 「测试是外挂的验证工具」——否。intent 原文把 automated test 与 users 并列为平等驱动者；测试适配器是一等公民。
