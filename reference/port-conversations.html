端口对话速查

端口 = 一段有目的的对话。粒度与命名的压缩版，供评审时快速对照。 出处：Cockburn 2005。

一句话

端口标识一段有目的的对话；协议由对话目的给出；技术只住在适配器里。 典型应用只有 2–4 个端口。

命名

风格示例判定
目的句式（Cockburn） ForStoringOrders · ForCalculatingTaxes · ForNotifyingShipments 合格
角色/意图（DDD 惯用） OrderRepository · EventPublisher · CustomerNotifier 合格
技术贴牌 MySqlPort · KafkaProducerPort · RedisGateway（当端口名时） 不合格——技术名留给适配器类

粒度两极端

极端样子问题
过细 每个用例一个端口（上百个） 对话被拆成函数清单，丢失「目的」聚合
过粗 左一个 API、右一个 API 说不清系统在跟外界谈哪几段话
合适 按自然对话：2–4 段 换技术后对话仍在 = 边界站得住

Cockburn 的「自然端口」例

  • 天气预警：采集气象 · 管理 · 通知订阅者 · 订阅者数据（4）
  • 咖啡机：用户 · 配方库 · 出货装置 · 投币盒（4）
  • 最简计税：算税（驱动）· 取税率（被驱动）（2）

30 秒自检

  1. 换掉一种外部技术，这段对话还在吗？——不在 → 你把适配器抬成了端口。
  2. 一个端口上能否想象出 ≥2 个适配器（含测试替身）？——不能 → 边界可能贴技术贴太紧。
  3. 技术词出现在适配器类名上合格（SqlOrderRepository）；出现在端口名上不合格。 把两个数据库品牌拆成两个「端口」= 把适配器抬成了端口。
  4. 通知型（不要答案）仍是被驱动端口——merely notify。
