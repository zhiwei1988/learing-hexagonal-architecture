# 第 5 课全对：能区分架构观察轴，并用依赖与 S2 证据做评审

第 5 课（对比评审）：回顾 2/2、练一练 8/8、试一试 5/5。用户已能区分 Hexagonal 的
「参与者对话」、Clean 的「政策/机制圈层」、Onion 的「领域模型居中」与传统分层的「职责堆栈」，
并能避免用目录名、架构图或流派标签代替可执行证据。

## Evidence

- **回顾 2/2**：主目标=S2≠S5、Cucumber/FIT+右侧 mock=S2 均答对；
  learning-records/0005 的「测试线束 vs 生产驱动」与「目标阶段」两个软点结案。
- **练一练 8/8**：四种架构视图的观察轴与原始术语全部分清，未把三者误当成互斥选项。
- **试一试 5/5**：能以源码依赖向内 + S2 孤立回归作为评审证据；同时能接受简单 CRUD
  继续使用清楚的传统分层，未落入「接口越多越成熟」的过度设计。

## Implications

- Mission 的「能讲清、能评审」两项已取得测验证据；课程可进入最后的真实项目迁移弧。
- 后续不再抽象讲架构流派。以用户现有 C++ 项目为现场，先收集领域、规模、痛点，再做
  接缝识别 → 特征测试 → 抽第一个端口。
- 迁移评审沿用本课证据顺序：具体业务变化 → 编译依赖 → 边界对话 → 替换测试 → S2。
